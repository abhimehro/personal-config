## 2026-04-27 - Command Injection Risk via eval in dynamic variables
**Vulnerability:** Command Injection ([CWE-78](https://cwe.mitre.org/data/definitions/78.html)) risk existed in `configs/.config/mole/lib/core/app_protection.sh` and `configs/.config/mole/lib/core/log.sh`. Functions used `eval` to assign and read variables dynamically based on function arguments without validating the variable name (e.g. `eval "$var_name=\"\$regex\""`, `eval "$varname=$ts"`). If an attacker could control the variable name passed to these functions, they could inject arbitrary bash commands.
**Learning:** Using `eval` to mimic pass-by-reference variable assignment in shell scripts exposes the script to command injection vulnerabilities if the variable name is not strictly validated. Furthermore, using `declare -n` for arrays breaks backward compatibility with Bash 3.2 (macOS default), thus it's safer to stick to validated `eval` where bash 3.2 compatibility is needed for array indirect expansion, or safely use `printf -v` and indirect variables `${!var}` with strict alphanumeric regex validation.
**Prevention:** Strictly validate the dynamically passed variable name against `^[a-zA-Z_][a-zA-Z0-9_]*$` before evaluation to prevent Command Injection (CWE-78). Replace `eval` with `printf -v` and `${!var}` where compatible.
