### New patterns observed

- **Pattern A (Lesson 0x in `tasks/lessons.md` below): JSON-encoded source-file blob.** A bot agent committed `src/modules/media_analyzer.py` as a single-line JSON-string-encoded blob (literal `\\n` and `\\"` escapes at offset 0), corrupting the entire file and breaking pytest collection. Detection: when a CI infra failure is a `SyntaxError` at line 1, the file might be a stringified blob — try `codecs.decode(content, 'unicode_escape')` to inspect the intended content. Even when the decode produces valid Python, the *content* may be a regression — preferred fix is revert to the parent commit.
- **Pattern B (Lesson 0y below): "Cleanup" PRs that truncate journal/append-only files.** PR #820 truncated `.jules/bolt.md` from 131 lines to 11 alongside its valid `.py` optimization. Salvage by taking only the source-code changes and appending (not replacing) journal entries.

- **Pattern D (Lesson 0aa below): `update-branch` HTTP 422 means a real conflict, not a stale base.** GitHub's `PUT /repos/{owner}/{repo}/pulls/{pull_number}/update-branch` returns 422 when there are merge conflicts that the API cannot auto-resolve. When this happens, the salvage path is to apply just the unique files from the PR onto a fresh branch rather than try to re-rebase the original branch.
- **Pattern E:** Salvage PRs are the right pattern when the original branch is too noisy to rebase cleanly. Cherry-pick or `git checkout pr_branch -- file/path` the unique work onto a fresh branch from main, preserve provenance via the cherry-pick author, and close the original PR with an explicit cross-link to the salvage PR.
